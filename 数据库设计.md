- **用户数据分离** ：严格遵循 Supabase 的标准模式，将用户的认证信息保留在 `auth.users` 表中，而将应用的公共用户信息（如订阅状态、头像等）存储在 `public.profiles` 表中，通过用户ID进行关联。
- **性能与并发**：通过创建聚合表 (例如 `daily_usage`) 来跟踪用量，而不是为每次生成都插入一条新记录。
- **关系完整性**：使用外键约束来确保数据的一致性和完整性。

### **表: `profiles`**

- **用途**: 存储与 `auth.users` 关联的用户特定信息，主要用于管理订阅状态和API使用配额。
- **列定义**:
| 列名 | 数据类型 | 描述与约束 |
| :--- | :--- | :--- |
| `id` | `UUID` | **主键 (Primary Key)**。与 `auth.users.id` 形成一对一关联 (`FOREIGN KEY`)。用户删除时级联删除此行 (`ON DELETE CASCADE`)。 |
| `updated_at` | `TIMESTAMPTZ` | 记录最后更新时间，可用于调试。 |
| `paddle_customer_id`| `TEXT` | 存储来自 Paddle 支付系统的客户ID。`UNIQUE` 约束。 |
| `status` | `subscription_status` | 用户的订阅状态。枚举类型，值为 `'free'`, `'active'`, `'canceled'`, `'past_due'`。默认为 `'free'`。`NOT NULL`。 |
| `active_price_id` | `TEXT` | 如果用户是付费会员，则存储其订阅的 Paddle 价格ID（用于区分月度/年度）。 |
| `generation_count` | `INT` | 用户当天已使用的歌词**生成**次数。默认为 `0`。`NOT NULL`。 |
| `rewrite_count` | `INT` | 用户当天已使用的**AI辅助重写**次数。默认为 `0`。`NOT NULL`。 |
| `usage_last_reset`| `DATE` | 上次重置 `generation_count` 和 `rewrite_count` 的日期。默认为当天。`NOT NULL`。 |
- **核心逻辑与关系**:
    - **用户关联**: 每个 `profiles` 记录都必须对应 `auth.users` 中的一个用户。
    - **用量检查**: 后端在执行 AI 生成或重写操作前，必须查询此表：
        1. 检查 `usage_last_reset` 是否是今天。
        2. 如果不是，将 `generation_count` 和 `rewrite_count` 重置为0，并将 `usage_last_reset` 更新为今天。
        3. 检查相应计数是否已达到用户等级的上限（免费1次，付费25次）。
    - **支付集成**: 当 Paddle Webhook 通知订阅成功时，后端通过 `paddle_customer_id` 找到对应用户，并更新其 `status` 和 `active_price_id`。
- **行级安全 (RLS)**: 用户只能读取和更新自己的 `profiles` 记录。

### **2. 表: `generations`**

- **用途**: 记录用户的每一次歌词生成历史，包括所有输入参数和最终输出。这是应用的核心数据表。
- **列定义**:
| 列名 | 数据类型 | 描述与约束 |
| :--- | :--- | :--- |
| `id` | `BIGINT` | **主键 (Primary Key)**。自增ID。 |
| `user_id` | `UUID` | **外键 (Foreign Key)**，关联到 `public.profiles.id`。`NOT NULL`。 |
| `created_at` | `TIMESTAMPTZ` | 记录的创建时间。默认为 `NOW()`。`NOT NULL`。 |
| `language` | `TEXT` | 输入参数：语言。 |
| `music_style` | `TEXT` | 输入参数：音乐风格。 |
| `music_theme` | `TEXT` | 输入参数：音乐主题。 |
| `length_option` | `TEXT` | 输入参数：长度。 |
| `lyric_style` | `TEXT` | 输入参数：歌词风格。 |
| `intent_or_request`| `TEXT` | 输入参数：创作意图/其他要求。 |
| `artist_style` | `TEXT` | 输入参数：参考歌手风格。 |
| `emotion_intensity`| `SMALLINT`| 输入参数：情感强度 (例如 1-100)。 |
| `rhyme_requirement`| `TEXT` | 输入参数：韵脚要求。 |
| `song_structure`| `TEXT` | 输入参数：歌曲结构 (e.g., "Verse-Chorus")。 |
| `paragraph_length`| `TEXT` | 输入参数：段落长度描述。 |
| `bpm` | `SMALLINT`| 输入参数：BPM。 |
| `generated_lyrics`| `TEXT` | AI生成的完整歌词文本。 |
| `model_used` | `TEXT` | 记录使用的模型（例如 "basic" 或 "pro"）。 |
| `is_favorited` | `BOOLEAN`| 用户是否收藏了此条记录。默认为 `FALSE`。`NOT NULL`。 |
- **核心逻辑与关系**:
    - **所有权**: 每条记录都属于一个用户 (`user_id`)。
    - **历史记录页**: 前端通过查询此表（`WHERE user_id = 'current_user_id' ORDER BY created_at DESC`）来展示用户的生成历史。
    - **收藏功能**: 用户点击收藏时，前端将对应记录的 `is_favorited` 更新为 `TRUE`。
    - **数据生命周期**: 一个每日运行的 Cron Job 会自动删除所有 `is_favorited = FALSE` 且 `created_at` 早于24小时的记录。
- **行级安全 (RLS)**: 用户只能对自己创建的 `generations` 记录进行增、删、改、查操作。

### **3. 表: `categories`**

- **用途**: 存储博客文章的分类。
- **列定义**:
| 列名 | 数据类型 | 描述与约束 |
| :--- | :--- | :--- |
| `id` | `BIGINT` | **主键 (Primary Key)**。自增ID。 |
| `created_at` | `TIMESTAMPTZ`| 创建时间。默认为 `NOW()`。 |
| `name` | `TEXT` | 分类名称（例如 "嘻哈与说唱"）。`UNIQUE` 且 `NOT NULL`。 |
| `slug` | `TEXT` | 用于URL的唯一标识（例如 "hip-hop-rap"）。`UNIQUE` 且 `NOT NULL`。 |
| `seo_title` | `TEXT` | 用于分类归档页的 `<title>` 标签。 |
| `meta_description`| `TEXT` | 用于分类归档页的 `<meta name="description">` 标签。 |
- **核心逻辑与关系**:
    - **一对多**: 一个 `categories` 记录可以关联到多个 `posts` 记录。
    - **后台管理**: 在后台管理系统中，应对此表进行增删改查。
- **行级安全 (RLS)**: 对所有人开放读取权限。写入和修改操作应由后端服务角色完成。

### **4. 表: `posts`**

- **用途**: 存储每一篇博客文章及其SEO元数据。
- **列定义**:
| 列名 | 数据类型 | 描述与约束 |
| :--- | :--- | :--- |
| `id` | `BIGINT` | **主键 (Primary Key)**。自增ID。 |
| `created_at` | `TIMESTAMPTZ`| 创建时间。默认为 `NOW()`。 |
| `category_id` | `BIGINT` | **外键 (Foreign Key)**，关联到 `public.categories.id`。`NOT NULL`。 |
| `title` | `TEXT` | 文章标题。将用作 `<h1>` 和默认的 `<title>`。`NOT NULL`。 |
| `content` | `TEXT` | 文章正文，存储由富文本编辑器生成的HTML。 |
| `status` | `post_status` | 文章状态。枚举类型，值为 `'draft'` 或 `'published'`。默认为 `'draft'`。 |
| `slug` | `TEXT` | 用于文章URL的唯一标识（例如 "how-to-write-rap-lyrics"）。`UNIQUE` 且 `NOT NULL`。 |
| `seo_title` | `TEXT` | （可选）覆盖默认的页面 `<title>` 标签。 |
| `meta_description`| `TEXT` | 页面的 `<meta name="description">` 标签。 |
- **核心逻辑与关系**:
    - **分类归属**: 每篇文章 (`posts`) 必须属于一个分类 (`categories`)。
    - **前端展示**: 前端只查询 `status = 'published'` 的文章。
    - **页面生成**: 在 Next.js 中，`slug` 用于生成静态页面路径 (`/blog/[slug]`)。`title`, `content`, `seo_title`, `meta_description` 用于填充页面内容和元数据。
- **行级安全 (RLS)**: `status` 为 `'published'` 的文章对所有人开放读取权限。写入和修改操作应由后端服务角色完成。

每个24小时删除未收藏的记录已经实现不需要额外完成。
